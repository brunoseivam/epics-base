45 ### \*_API macros in EPICS headers
46 
47 Internally, the Com and ca libraries now express dllimport/export (Windows)
48 and symbol visibility (GCC) using library-specific macros (eg. `LIBCOM_API`)
49 instead of the macros `epicsShareFunc`, `epicsShareClass`, `epicsShareDef` etc.
50 that are defined in the `shareLib.h` header.
51 This change may affect some user code which uses the `epicsShare*` macros
52 without having explicitly included the `shareLib.h` header themselves.
53 Such code should be changed to include `shareLib.h` directly.
54 
55 A new helper script `makeAPIheader.pl` and build rules to generate a
56 library-specific `*API.h` header file has been added. Run `makeAPIheader.pl -h`
57 for information on how to use this in your own applications, but note that the
58 resulting sources will not be able to be compiled using earlier versions of
59 EPICS Base.

61 ### IOCsh usage messages
62 
63 At the iocShell prompt `help <cmd>` now prints a descriptive usage message
64 for many internal IOCsh commands in addition to the command parameters.
65 Try `help *` to see all commands, or a glob pattern such as `help db*` to see
66 a subset.
67 
68 External code may provide usage messages when registering commands using a
69 new `const char *usage` member of the `iocshFuncDef` structure.
70 The `iocsh.h` header also now defines a macro `IOCSHFUNCDEF_HAS_USAGE` which
71 can be used to detect Base versions that support this feature at compile-time.

73 ### Variable names in RELEASE files
74 
75 `configure/RELEASE` files are parsed by both GNUmake and the `convertRelease.pl`
76 script. While GNUmake is quite relaxed about what characters may be used in a
77 RELEASE variable name, the `convertRelease.pl` script parser has only recognized
78 variable names that match the Perl regular expression `\w+`, i.e. upper and
79 lower-case letters, digits and underscore characters.
80 
81 The script has been modified so now RELEASE variable names must start with a
82 letter or underscore, and be followed by any number of letters, digits,
83 underscore or hyphen characters, matching the regular expression
84 `[A-Za-z_][A-Za-z_0-9-]*`. The hyphen character `-` was not previously allowed
85 and if used would have prevented a build from finding include files and
86 libraries in any module using that in its RELEASE variable name.
87 
88 This change does disallow names that start with a digit which used to be
89 allowed, but hopefully nobody has been relying on that ability. The regular
90 expression used for names can be found in the file `src/tools/EPICS/Release.pm`
91 and can be adjusted locally if necessary.

94

102 which may be necessary to see any error messages it may emit.

110 ### Record types publish dset's
111 
112 The record types in Base now define their device support entry table (DSET)
113 structures in the record header file. While still optional, developers of
114 external support modules are encouraged to start converting their code to use
115 the record's new definitions instead of the traditional approach of copying the
116 structure definitions into each source file that needs them. By following the
117 instructions below it is still possible for the converted code to build and
118 work with older Base releases.
119 
120 This would also be a good time to modify the device support to use the type-safe
121 device support entry tables that were introduced in Base-3.16.2 -- see
122 [#type-safe-device-and-driver-support-tables](this entry below) for the
123 description of that change, which is also optional for now.
124 
125 Look at the aiRecord for example. Near the top of the generated `aiRecord.h`
126 header file is a new section that declares the `aidset`:
127 
128 ```C
129 /* Declare Device Support Entry Table */
130 struct aiRecord;
131 typedef struct aidset {
132  dset common;
133  long (*read_ai)(struct aiRecord *prec);
134  long (*special_linconv)(struct aiRecord *prec, int after);
135 } aidset;
136 #define HAS_aidset
137 ```
138 
139 Notice that the common members (`number`, `report()`, `init()`, `init_record()`
140 and `get_ioint_info()` don't appear directly but are included by embedding the
141 `dset common` member instead. This avoids the need to have separate definitions
142 of those members in each record dset, but does require those members to be
143 wrapped inside another set of braces `{}` when initializing the data structure
144 for the individual device supports. It also requires changes to code that
145 references those common members, but that code usually only appears inside the
146 record type implementation and very rarely in device supports.
147 
148 An aiRecord device support that will only be built against this or later
149 versions of EPICS can now declare its dset like this:
150 
151 ```C
152 aidset devAiSoft = {
153  { 6, NULL, NULL, init_record, NULL },
154  read_ai, NULL
155 };
156 epicsExportAddress(dset, devAiSoft);
157 ```
158 
159 However most device support that is not built into EPICS itself will need to
160 remain compatible with older EPICS versions, which is why the ai record's header
161 file also declares the preprocessor macro `HAS_aidset`. This makes it easy to
162 define the `aidset` in the device support code when it's needed, and not when
163 it's provided in the header:
164 
165 ```C
166 #ifndef HAS_aidset
167 typedef struct aidset {
168  dset common;
169  long (*read_ai)(aiRecord *prec);
170  long (*special_linconv)(aiRecord *prec, int after);
171 } aidset;
172 #endif
173 aidset devAiSoft = {
174  { 6, NULL, NULL, init_record, NULL },
175  read_ai, NULL
176 };
177 epicsExportAddress(dset, devAiSoft);
178 ```
179 
180 The above `typedef struct` declaration was copied directly from the new
181 aiRecord.h file and wrapped in the `#ifndef HAS_aidset` conditional.
182 
183 This same pattern should be followed for all record types except for the lsi,
184 lso and printf record types, which have published their device support entry
185 table structures since they were first added to Base but didn't previously embed
186 the `dset common` member. Device support for these record types therefore can't
187 use the dset name since the new definitions are different from the originals and
188 will cause a compile error, so this pattern should be used instead:
189 
190 ```C
191 #ifndef HAS_lsidset
192 struct {
193  dset common;
194  long (*read_string)(lsiRecord *prec);
195 }
196 #else
197 lsidset
198 #endif
199 devLsiEtherIP = {
200  {5, NULL, lsi_init, lsi_init_record, get_ioint_info},
201  lsi_read
202 };
203 ```

253 ### Relocatable Builds
254 
255 Allows built trees to be copied or moved without invalidating RPATH entires.
256 
257 The `LINKER_USE_RPATH` Makefile variable (see `configure/CONFIG_SITE`) may be
258 set to `YES`, `NO`, and a new third option `ORIGIN`. This is limited to
259 targets using the ELF executable format (eg. Linux).
260 
261 When `LINKER_USE_RPATH=ORIGIN`, the variable `LINKER_ORIGIN_ROOT` is set to
262 one of the parents of the build directory. Any libraries being linked
263 to which are found under this root will have a relative RPATH entry.
264 Other libraries continue to result in absolute RPATH entries.
265 
266 An effect of this might change a support library from being linked with
267 `-Wl,-rpath /build/epics-base/lib/linux-x86`
268 to being linked with
269 `-Wl,-rpath \$ORIGIN/../../../epics-base/lib/linux-x86`
270 if the support module directory is `/build/mymodule`
271 and `LINKER_ORIGIN_ROOT=/build`.
272 
273 The API functions `epicsGetExecDir()` and `epicsGetExecName()` are also
274 added to `osiFileName.h` to provide runtime access to the directory or
275 filename of the executable with which the process was started.
276 
277 ### Decouple LINKER_USE_RPATH and STATIC_BUILD
278 
279 Previously, setting `STATIC_BUILD=NO` implied `LINKER_USE_RPATH=NO`.
280 This is no longer the case. Setting `LINKER_USE_RPATH=YES` will
281 always emit RPATH entries. This was found to be helpful when linking
282 against some 3rd party libraries which are only available as shared objects.

284 ### Channel Access Security: Check Hostname Against DNS
285 
286 Host names given in a `HAG` entry of an IOC's Access Security Configuration
287 File (ACF) have to date been compared against the hostname provided by the CA
288 client at connection time, which may or may not be the actual name of that
289 client. This allows rogue clients to pretend to be a different host, and the
290 IOC would believe them.
291 
292 An option is now available to cause an IOC to ask its operating system to look
293 up the IP address of any hostnames listed in its ACF (which will normally be
294 done using the DNS or the `/etc/hosts` file). The IOC will then compare the
295 resulting IP address against the client's actual IP address when checking
296 access permissions at connection time. This name resolution is performed at
297 ACF file load time, which has a few consequences:
298 
299  1. If the DNS is slow when the names are resolved this will delay the process
300 of loading the ACF file.
301 
302  2. If a host name cannot be resolved the IOC will proceed, but this host name
303 will never be matched.
304 
305  3. Any changes in the hostname to IP address mapping will not be picked up by
306 the IOC unless and until the ACF file gets reloaded.
307 
308 Optionally, IP addresses may be added instead of, or in addition to, host
309 names in the ACF file.
310 
311 This feature can be enabled before `iocInit` with
312 
313 ```
314  var("asCheckClientIP",1)
315 ```
316 
317 or with the VxWorks target shell use
318 
319 ```C
320  asCheckClientIP = 1
321 ```

323 ### New and modified epicsThread APIs
324 
325 #### `epicsThreadCreateOpt()`
326 
327 A new routine `epicsThreadCreateOpt()` is an alternative to
328 `epicsThreadCreate()` which takes some arguments via a structure (`struct
329 epicsThreadOpts`) to allow for future extensions.
330 
331 ```C
332  typedef struct epicsThreadOpts {
333  unsigned int priority;
334  unsigned int stackSize;
335  unsigned int joinable;
336  } epicsThreadOpts;
337  #define EPICS_THREAD_OPTS_INIT { \
338  epicsThreadPriorityLow, epicsThreadStackMedium, 0}
339  epicsThreadId epicsThreadCreateOpt(const char * name,
340  EPICSTHREADFUNC funptr, void * parm, const epicsThreadOpts *opts);
341 ```
342 
343 The final `opts` parameter may be `NULL` to use the default values of thread
344 priority (low) and stack size (medium). Callers wishing to provide alternative
345 settings for these thread options or to create a joinable thread (see below)
346 should create and pass in an `epicsThreadOpts` structure as shown below.
347 Always initialize one of these structures using the `EPICS_THREAD_OPTS_INIT`
348 macro to ensure that any additional fields that get added in the future are
349 set to their default values.
350 
351 ```C
352  void startitup(void) {
353  epicsThreadOpts opts = EPICS_THREAD_OPTS_INIT;
354  epicsThreadId tid;
355 
356  opts.priority = epicsThreadPriorityMedium;
357  tid = epicsThreadCreateOpt("my thread", &threadMain, NULL, &opts);
358  }
359 ```
360 
361 C or C++ Code that also needs to build on earlier versions of Base can use
362 `#ifdef EPICS_THREAD_OPTS_INIT` to determine whether the
363 `epicsThreadCreateOpt()` API is available on this Base version.
364 
365 #### Thread stack sizes
366 
367 The `stackSize` member of the `epicsThreadOpts` structure and the equivalent
368 parameters to the `epicsThreadCreate()` and `epicsThreadMustCreate()` routines
369 can now be passed either one of the `epicsThreadStackSizeClass` enum values or
370 a value returned from the `epicsThreadGetStackSize()` routine.
371 
372 #### `epicsThreadMustJoin()`
373 
374 If the new `joinable` flag of an `epicsThreadOpts` structure is non-zero (the
375 default value is zero), the new API routine `epicsThreadMustJoin()` *must* be
376 called with the thread's `epicsThreadId` when/after the thread exits, to free
377 up thread resources. This function will block until the thread's main function
378 has returned, allowing the parent to wait for its child thread. The child's
379 `epicsThreadId` will no longer be valid and should not be used after the
380 `epicsThreadMustJoin()` routine returns.
381 
382 A thread that was originally created with its joinable flag set may itself
383 call `epicsThreadMustJoin()`, passing in its own epicsThreadId. This marks the
384 thread as no longer being joinable, so it will then free the thread resources
385 itself when its main function returns. The `epicsThreadId` of a thread that is
386 not joinable gets invalidated as soon as its main function returns.

435

437 branch between official releases would have shown the version number of the
438 previous release, followed by a -DEV suffix, for example 7.0.2.1-DEV.

520 ## EPICS Release 7.0.1.1
521 
522 ### Changed SIML failure behavior
523 
524 A failure when fetching the simulation mode through `SIML` will not put the
525 record into INVALID alarm state anymore. Instead, as long as the record's
526 current alarm severity (`SEVR`)is `NO_ALARM`, its alarm status (`STAT`) will be
527 set to `LINK_ALARM` without increasing the severity. This allows clients to get
528 some notification of a failing or bad `SIML` link without otherwise affecting
529 record processing.
530 
531 ### `dbVerify()` has been restored to dbStaticLib
532 
533 This routine was removed in Base-3.16.1 but has been reimplemented in this
534 release by special request. Note that the error message strings that it
535 returns when verification fails have changed, but are still designed for
536 display to the user.
537 
538 ### Simulation mode improvements
539 
540 Records that support simulation mode have two new fields, `SSCN` (Simulation
541 Scan Mode) and `SDLY` (Simulation Delay). `SSCN` is a menu field that provides
542 an alternate value for the `SCAN` field to be used while the record is in
543 simulation mode. This is especially useful for I/O scanned records, for which
544 simulation mode was not working at all. Setting `SDLY` to a positive value
545 makes the record process asynchronously in simulation mode, with the second
546 stage processing happening after the specified time (in seconds).
547 
548 ### Extend the dbServer API with init/run/pause/stop methods
549 
550 This change permits IOCs to be built that omit the CA server (RSRV) by
551 removing its registrar entry which is now provided in the new `rsrv.dbd` file.
552 Other server layers can be built into the IOC (alongside RSRV or in place of
553 it) by registering them in a similar manner. The dbServer API is documented
554 with Doxygen comments in the header file.
555 
556 Specific IOC server layers can be disabled at runtime by adding their name to
557 the environment variable `EPICS_IOC_IGNORE_SERVERS` (separated by spaces if more
558 than one should be ignored).

560 ### Grand source-code reorganization
561 
562 EPICS 7.0.1 contains the IOC Database, RSRV server and the Channel Access
563 client code from EPICS Base 3.16.1 along with all the original record types
564 and soft device support, but GDD and the Portable Channel Access Server have

569 been split into 4 separate Git repositories. External modules should build
570 against this new structure with little or no changes needed, except that some
571 allowance may be needed for the merging of the V4 modules.

574 than is currently available, but as developers we generally much prefer to
575 write code than documentation. Send questions to the tech-talk mailing list

807

811  #else

929 ### Extensible Link Types
930 
931 A major new feature introduced with this release of EPICS Base is an
932 Extensible Link Type mechanism, also known as Link Support or JSON Link Types.
933 This addition permits new kinds of link I/O to be added to an IOC in a similar
934 manner to the other extension points already supported (e.g. record, device
935 and driver support).
936 
937 A new link type must implement two related APIs, one for parsing the JSON
938 string which provides the link address and the other which implements the link
939 operations that get called at run-time to perform I/O. The link type is built
940 into the IOC by providing a new `link` entry in a DBD file.
941 
942 #### New Link Types Added
943 
944 This release contains two new JSON link types, `const` and `calc`:
945 
946  * The `const` link type is almost equivalent to the old CONSTANT link type
947 with the updates described below to accept arrays and strings, except that
948 there is no need to wrap a scalar string constant inside array brackets since
949 a constant string will never be confused with a PV name.
950 
951  * The `calc` link type allows CALC expressions to be used to combine
952 values from other JSON links to produce its value. Until additional JSON link
953 types are created though, the `calc` link type has little practical utility as
954 it can currently only fetch inputs from other `calc` links or from `const`
955 links.
956 
957 ```
958  field(INP, {calc:{expr:"A+B+1",
959  args:[5, # A
960  {const:6}] # B
961  }
962  }
963  )
964 ```
965

974 #### Support Routine Modifications for Extensible Link Types
975 
976 For link fields in external record types and soft device support to be able
977 to use the new link types properly, various changes are required to utilize
978 the new Link Support API as defined in the dbLink.h header file and outlined
979 below. The existing built-in Database and Channel Access link types have been
980 altered to implement the link APIs, so will work properly after these
981 conversions:
982 
983  * Make all calls to `recGblInitConstantLink()` unconditional on the link
984 type, i.e. change this code:
985 
986 ```C
987  if (prec->siml.type == CONSTANT) {
988  recGblInitConstantLink(&prec->siml, DBF_USHORT, &prec->simm);
989  }
990 ```
991 
992  into this:
993 
994 ```C
995  recGblInitConstantLink(&prec->siml, DBF_USHORT, &prec->simm);
996 ```
997 
998  Note that `recGblInitConstantLink()` still returns TRUE if the field was
999  successfully initialized from the link (implying the link is constant).
1000  This change will work properly with all Base releases currently in use.
1001 
1002  * Code that needs to identify a constant link should be modified to use
1003 the new routine `dbLinkIsConstant()` instead, which returns TRUE for constant
1004 or undefined links, FALSE for links whose `dbGetLink()` routine may return
1005 different values on different calls. For example this:
1006 
1007 ```C
1008  if (prec->dol.type != CONSTANT)
1009 ```
1010 
1011  should become this:
1012 
1013 ```C
1014  if (!dbLinkIsConstant(&prec->dol))
1015 ```
1016 
1017  When the converted software is also required to build against older versions
1018  of Base, this macro definition may be useful:
1019 
1020 ```C
1021  #define dbLinkIsConstant(lnk) ((lnk)->type == CONSTANT)
1022 ```
1023 
1024  * Any code that calls dbCa routines directly, or that explicitly checks if
1025 a link has been resolved as a CA link using code such as
1026 
1027 ```C
1028  if (prec->inp.type == CA_LINK)
1029 ```
1030 
1031  will still compile and run, but will only work properly with the old CA link
1032  type. To operate with the new extensible link types such code must be
1033  modified to use the new generic routines defined in dbLink.h and should
1034  never attempt to examine or modify data inside the link. After conversion
1035  the above line would probably become:
1036 
1037 ```C
1038  if (dbLinkIsVolatile(&prec->inp))
1039 ```
1040 
1041  A volatile link is one like a Channel Access link which may disconnect and
1042  reconnect without notice at runtime. Database links and constant links are
1043  not volatile; unless their link address is changed they will always remain
1044  in the same state they started in. For compatibility when building against

1046 
1047 ```C
1048  #define dbLinkIsVolatile(lnk) ((lnk)->type == CA_LINK)

1055 
1056 ```C

1058 ```
1059 
1060  which should become:
1061 
1062 ```C
1063  stat = dbIsLinkConnected(plink);
1064 ```
1065 
1066  Similar changes should be made for calls to the other dbCa routines.
1067 
1068  * A full example can be found by looking at the changes to the calcout

1073

1098  field(NELM, 10)

1118 ### Database Parsing of "Relaxed JSON" Values
1119 
1120 A database file can now provide a "relaxed JSON" value for a database field
1121 value or an info tag. Only a few field types can currently accept such values,
1122 but the capability is now available for use in other places in the future.
1123 When writing to a JSON-capable field at run-time however, only strictly
1124 compliant JSON may be used (the dbStaticLib parser rewrites relaxed JSON
1125 values into strict JSON before passing them to the datase for interpretation,
1126 where the strict rules must be followed).
1127 
1128 "Relaxed JSON" was developed to maximize compatibility with the previous
1129 database parser rules and reduce the number of double-quotes that would be
1130 needed for strict JSON syntax. The parser does accept strict JSON too though,
1131 which should be used when machine-generating database files. The differences
1132 are:
1133 
1134  * Strings containing only the characters `a-z A-Z 0-9 _ - + .` do not have to
1135 be enclosed in double-quote characters.
1136 
1137  * The above rule applies to map keys as well as to regular string values.
1138 
1139  * The JSON keywords `null`, `true` and `false` (all lower-case) will be
1140 recognized as keywords, so they must be quoted to use any of these single words
1141 as a string.
1142 
1143  * Comments may be used, introduced as usual by the `#` character and extending
1144 to the end of the line.
1145
